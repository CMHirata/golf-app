Background & Strategic Context

The app has grown to a level of complexity — 7 side games, press hierarchies, subset pickers, handicap modes, multi-player match formats — where informal "build as you go" development stops scaling. Two parallel investments are now warranted:

  1. Specification Contracts — written rules each game and subsystem must conform to, enabling parallel developer work and making bugs easier to identify and fix
  2. Code Architecture Refactor — breaking large files into independently maintainable modules, reducing the largest file from 1,627 lines to ~250
Recommended sequence:

Contracts → Code split → Fix & test. Writing contracts first defines what "correct" means before any bugs are fixed. The code split then makes fixing easier by isolating each game into its own file.

Part 1 — Specification Contracts

The existing Match_Nassau_Contract.md is the right model. The approach is one contract per logical domain, not one per game. Nine contracts are recommended:

Part 3 — Recommended Sequence

Key principle:

Contracts → Code split → Fix & test. Don't fix bugs in the 1,627-line file then split — do careful work in hard-to-navigate files. Split first (mechanical, low-risk), then fix in the smaller files.

Two parallel tracks

Track A — Contracts + architecture (do first, unlocks everything)

1
Write contracts for already-working games

Nassau/Match (expand existing), Skins, Stableford, Stroke Play. No code changes.

2
Do the code split

Pure extract-and-re-export. Zero logic changes. Start with MatchNassauTable — most complex, highest value to isolate.

3
Fix and test in the smaller files

Each game table is now independently editable. Bugs are easier to scope, verify, and hand off.

Track B — Rules decisions for broken/incomplete games (can run in parallel)

A
Decide Sixes press behavior

Write Sixes contract → fix Sixes press wiring in the extracted SixesTable.jsx

B
Decide handicap dot display strategy

Write that section of the Handicap contract → implement in ScoreGrid.jsx

C
Resolve remaining open items

stableford_players in roundLib, SkinsTable subset display, matches[] in HistoryPage — each now has a clear home file
